The panel now queues the target file, waits until the diff pane is actually visible, and scrolls the nearest vertical container with explicit offset math instead of relying on bare `scrollIntoView`.

The panel now renders a true directory tree with compact single-row file items, drops the redundant per-file `1f` badge, and reuses the existing backend `GitEntryFileSummary.diff` payload so each row keeps meaningful `+/-` stats without any server-side schema change. Diff-stream headers were also updated to wrap full paths instead of truncating them.
- A subsequent self-walk showed the first tree pass was still too fragmented because every single-child directory level consumed its own row. The tree now compresses linear directory chains (for example `src/components/git`) into one directory row and switches path wrapping to `overflow-wrap:anywhere` so long paths stay complete without the harsher per-character `break-all` behavior.
- A later data-semantics review found that untracked files were being serialized with synthetic `+0/-0` stats before their patch payload had actually been computed. The Git file model now distinguishes `ready` vs `loading` vs `unavailable` file-level diff stats, so the tree and diff headers render objective loading state instead of fake zero values.
- A subsequent UX correction split the old mixed `/git` screen into a commit list page and dedicated commit-detail routes (`/git/uncommitted` and `/git/commit/$hash`) so the information architecture now follows the same list/detail pattern used by `Changes`, including a top-left back action on detail.
- The File Tree was then rebuilt around a derived tree model instead of patch-query side effects: directory rows now aggregate descendant diff stats, ready rows no longer show completion checkmarks, and unresolved rows show only objective `loading`/`n/a` state.
- The tree visuals were also upgraded from simple left-border indentation to explicit guide rails and elbows so hierarchy reads as a real connected tree instead of a padded list.
- Detail-panel tree-to-diff navigation now measures the tabs strip height and applies the same offset to both `scroll-padding-top` and patch-card `scroll-margin-top`, preventing the target file header from landing under the tabs row after a tree jump.

The detail panel now keeps tree-jump scrolling pending until the target card is actually aligned, re-checks alignment after patch layout updates, and adds a regression test covering the second-pass scroll behavior.
- A later maintainer review found a second class of Git-detail defects that earlier passes had not modeled explicitly: the UI was still rendering `ready` zero-line diffs (`1f +0 -0`) as if they were valuable status, even though many of those rows were pure rename/archive moves with no readable line delta. The fix deliberately separates “computable diff” from “worth displaying diff”: zero-line ready stats and zero-file badges are now visually suppressed, single-file patch headers no longer repeat `1f`, pure rename cards use the destination path as the primary line plus a `from ...` secondary line, the commit-detail title/subtitle no longer truncate, and the File Tree is now a true collapsible tree with folder open/closed affordances instead of a permanently expanded pseudo-list.
- The latest commit-detail follow-up replaced the old selected-row model with a shared scroll-spy contract: the wide detail view now gives File Tree its own viewport-capped scroll area, the narrow detail view derives tree max-height from the live tabs-strip height plus balanced margin-block spacing, visible diff cards drive file-tree highlight intensity through shared visibility ratios, and Dashboard Git Snapshot commit rows now deep-link into the live Git detail route through the existing bottom-panel navigation controller.
- Validation for that follow-up now spans both levels of the platform: unit tests continue to cover tree reveal scrolling and deferred diff jumps, while a dedicated Storybook browser harness exercises the scroll/highlight layout contract in both wide and narrow modes without depending on React Query.
- A supporting Storybook infrastructure fix also moved the React-to-web-components bridge to mount only after the host element is connected, preventing DOM-observer/layout-sensitive stories from initializing against detached nodes.
- Real-device feedback then exposed three regressions in that first scroll-spy pass: narrow tabs introduced a second scrollbar, the wide File Tree viewport was constrained against the wrong box and could spill below the visible shell, and smooth tree reveal plus late scroll-root binding made the left tree feel like it was fighting manual scroll. The correction moved wide-tree sizing onto a measured viewport wrapper, restored narrow mode to natural document height, switched tree reveal to immediate container scrolling, and taught scroll-root discovery to bind against explicit overflow containers before they actually overflow.
- The same correction loop also found a Storybook environment defect rather than a product defect: the web-components preview had not imported the app's Tailwind entry, so scroll-layout stories were evaluating partially unstyled DOM. Storybook preview now imports `src/index.css`, and the browser story asserts stable viewport constraints while unit tests continue to cover the direct `scrollIntoView` reveal behavior.
- Post-release user validation then rejected the earlier “done” assumption: although layout regressions were fixed, the wide File Tree could still be yanked back to the active row because any later `IntersectionObserver` visibility event was still allowed to authorize tree reveal, even when the user was currently navigating the tree itself. The correction tightened the platform rule from “any new visible diff card may reveal the tree” to “only diff-owned navigation may authorize tree reveal”; tree-owned wheel/pointer/keyboard intent now suppresses reveal while preserving highlight updates, unit tests cover both suppression and positive diff-owned reveal, and the Storybook harness now exercises the exact tree-source guard that earlier stories had missed.
- A later worktree-lifecycle follow-up promoted “current project root disappeared” into a first-class platform signal instead of leaving it as watcher-local self-healing state. The reactive watcher runtime now reports `projectResidency`, the server derives `projectRecovery` from cached Git common-dir metadata, and the web shell auto-hands off to an existing default-branch worktree instance while preserving the current route across standalone and hosted modes.
- Final real-browser acceptance for the eviction-recovery pass now uses a temporary feature worktree as `projectDir`, deletes that directory on disk, and verifies that `/settings` auto-hands off from the dead worktree server to a surviving default-branch server while `/api/health` flips from the temporary worktree path back to the main repo path. That pass also exposed one more CLI-platform defect: when local handoff spawned a sibling instance through nested `pnpm run dev`, recovery could depend on script-runner semantics during an already-evicted session. The local workspace path now prefers the built `packages/cli/dist/cli.mjs` entry directly and only falls back to `pnpm run dev` when no local CLI build exists.

## Decisions Taken

- The first Git page only expands the current worktree in detail.
- Other worktrees stay at summary level inside this loop and expose a real switch action.
- Git detail uses a GitHub-style patch stream instead of a single-file diff viewer.
- Refresh preset/timing state is shared with Dashboard.
- Worktree switching must use child-instance handoff because the current watcher pool is process-global and single-project.
- Child worktree instances will be reused through a registry rather than spawned every time.
- `/git` remains excluded from the static route tree and static navigation.
- A canonical git-path comparison rule is now required on the server side because `git worktree list` can surface physical paths (for example `/private/var/...`) that differ from the caller's logical path (`/var/...`) while still referring to the same directory.
- Project-root eviction recovery must not re-query Git from the deleted worktree path. The valid source of truth after eviction is the cached absolute `git-common-dir`, and automatic recovery is intentionally limited to already-existing default-branch worktrees rather than creating new worktrees implicitly.
- Local sibling-instance recovery should prefer a direct CLI entrypoint over nested package-manager script indirection. In the monorepo that means using the built `packages/cli/dist/cli.mjs` when available, so recovery remains valid even while the source worktree has already been evicted.

## Divergence Notes

- No scope divergence from the approved plan so far.
- One unexpected implementation issue appeared during router testing: string-based path equality was not sufficient for worktree matching/current-worktree detection on macOS temp paths. This was resolved by adding a shared canonical git-path comparison helper instead of patching tests around the symptom.
- A second unexpected issue appeared during browser acceptance: `git worktree list --porcelain` can retain prunable entries whose directories are already gone. The UI was incorrectly offering `Switch worktree` for those dead entries. This was resolved by promoting `pathAvailable` into the shared worktree model, guarding the server mutation, and rendering stale rows as remove-only.
- A third follow-up issue appeared during Git panel validation: the original detail path was still eager-loading full patch payloads and the route itself forced a refresh on every mount, which made the page feel cold on each visit. This was resolved by introducing shell-vs-patch API split, version-keyed cache reuse, and removal of the route-mount refresh trigger.
- A fourth follow-up issue appeared during self-walk on a shell-managed local environment: process PATH inspection alone was insufficient to discover the user's `openspec` shim, which left the UI permanently blocked behind the CLI-setup gate even though `openspec` was available from the login shell. This was resolved by resolving bare `openspec` commands through the user's shell once during runner detection and then executing the resolved absolute path directly.
- A later validation issue appeared in Storybook rather than product scope: the web-components React bridge and the original detail story were not a reliable environment for scroll-observer verification. This was resolved in-platform by delaying React story mount until host connection and replacing the acceptance story with a deterministic scroll harness, without reopening intake or changing the approved Git UX scope.
- The latest validation issue came from maintainer/device feedback rather than new scope: the first wide/narrow scroll-spy rollout still had incorrect container-height and reveal semantics under real page composition. This was resolved in-platform by tightening the shared scroll-container rules and correcting the detail-panel layout, without changing the approved Git routing or dashboard handoff scope.
- A further validation issue came from the maintainer immediately after the first patch release: the browser story that had been accepted as sufficient was not actually modeling the wide-tree “manual left-nav vs observer-driven reveal” race, so the published build still allowed the tree to snap back under real use. This was resolved by reopening the loop in-place, updating the shared reveal-authorization rule, and replacing the overly synthetic Storybook assumption with a deterministic guard harness plus explicit unit coverage.
- The new project-root eviction recovery pass also exposed a platform-law bug during testing: the first implementation tried to refresh Git metadata after the worktree had already been deleted, which discarded the cached common-dir and made recovery impossible. The corrected rule is stricter: cache while the worktree is alive, and only refresh that cache on active residency, never after eviction has begun.
- A final browser-acceptance issue appeared in the local handoff runtime rather than the recovery algorithm itself: the first local workspace spawn path still depended on nested `pnpm run dev`, which proved too implicit under an evicted-session handoff. This was resolved by preferring the built local CLI entrypoint directly (`packages/cli/dist/cli.mjs`) and re-running the deletion scenario until the browser objectively moved from `http://localhost:62011/settings?_b=%2F` to `http://localhost:3102/settings?_b=%2F` with the project path restored to the main worktree.

## Loopback Triggers

- If Git patch extraction proves too slow or too large for selected entries, loop back to narrow the detail payload or add stricter per-entry limits.
- If child-instance handoff cannot be implemented safely from the current CLI process, loop back before introducing a partial or placeholder switch flow.
